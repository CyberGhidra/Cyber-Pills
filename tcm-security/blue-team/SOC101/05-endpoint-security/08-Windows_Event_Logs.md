# Windows Event Logs

---

## Overview

Windows Event Logs are one of the most valuable sources of information for both **defenders** (Blue Team) and **attackers performing enumeration** (Red Team). Every significant action on a Windows system — logins, service installs, user creation, process execution, policy changes — generates an event log entry. Understanding how to read and query these logs is a fundamental skill for incident response, threat hunting, and security auditing.

---

## 1. Event Log Categories

Windows organizes logs into several default channels, accessible via **Event Viewer** (`eventvwr.msc`):

| Log Name | Purpose |
|---|---|
| **Application** | Events generated by applications and programs (crashes, errors, info messages from software) |
| **Security** | Authentication events, privilege use, account management, policy changes — the most important log for security investigations |
| **Setup** | Events related to Windows installation and updates |
| **System** | Events generated by Windows system components and drivers (service starts/stops, hardware issues, kernel events) |
| **Forwarded Events** | Events collected from **remote machines** via Windows Event Forwarding (WEF) — useful in enterprise environments where logs are centralized |

> **Key point for defenders:** The **Security** log is the primary target for threat hunting. Always ensure it is enabled, properly sized, and ideally forwarded to a SIEM.

---

## 2. Event Severity Levels

Each event entry carries a **level** that indicates its nature:

| Level | Description |
|---|---|
| **Information** | A successful or routine operation (e.g., service started successfully) |
| **Warning** | Something unusual happened that may require attention but is not immediately critical |
| **Error** | A significant problem occurred (e.g., a service failed to start, a driver error) |
| **Critical** | A severe failure that requires immediate action (e.g., system about to crash) |
| **Audit Success** | *(Security log only)* An audited action was performed successfully (e.g., successful logon) |
| **Audit Failure** | *(Security log only)* An audited action was attempted and **failed** (e.g., failed logon — high value for detecting brute force) |

## 3. Event IDs

Every event has a numeric **Event ID** that identifies exactly what happened. Knowing the key IDs allows you to filter thousands of events down to exactly what matters.

### 3.1 Security Event IDs — Account Management

These events are generated when user accounts are created, modified, enabled, disabled, or deleted. All fall under the **Security** log.

| Event ID | Description | Why It Matters |
|---|---|---|
| **4720** | A user account was **created** | Attackers create backdoor accounts after gaining access |
| **4722** | A user account was **enabled** | A disabled account being re-enabled can indicate backdoor reactivation |
| **4723** | An attempt was made to **change a user's password** (by the user themselves) | Tracks password self-changes |
| **4724** | An attempt was made to **reset a user's password** (by an admin) | Admin-forced resets — look for unusual actors |
| **4725** | A user account was **disabled** | Could indicate legitimate admin action or an attacker disabling defenses |
| **4726** | A user account was **deleted** | Attackers may delete backdoor accounts to cover tracks after use |
| **4738** | A user account was **changed** (properties modified) | Membership changes, flag changes, description edits |

**Practical example — creating a backdoor account:**

```cmd
net user backdoor password /add
```

This command creates a new local user `backdoor` with the password `password`. On execution, Windows generates **Event ID 4720** in the Security log, recording:
- The new account name (`backdoor`)
- The **creator's** username and SID (who ran the command)
- The timestamp

**Practical example — deleting the backdoor account:**

```cmd
net user backdoor /delete
```

This generates **Event ID 4726**, recording the deletion and who performed it.

> **Attacker tradecraft:** Creating an account, using it, then deleting it is a common technique to establish temporary persistence. Both 4720 and 4726 should be monitored — creation *and* deletion of accounts are suspicious when performed by unexpected users.

---

### 3.2 Security Event IDs — Process & Audit

| Event ID | Description | Why It Matters |
|---|---|---|
| **4624** | A user account **successfully logged on** | Baseline for all logon activity; filter by Logon Type (2=interactive, 3=network, 10=remote interactive/RDP) |
| **4625** | A user account **failed to log on** | Repeated failures = brute force; single failure on privileged account = credential stuffing |
| **4688** | A **new process was created** | Records every process execution — attacker commands, malware execution, lateral movement tools. Requires audit policy to be enabled. |
| **1102** | The **Security audit log was cleared** | A major red flag — attackers clear logs to destroy evidence. This event itself is generated when clearing occurs. |

**Event ID 4688 — Process Creation:**  
When properly configured (via `auditpol` or Group Policy), every process that starts is logged with:
- The process name and full path
- The **parent process** (what spawned it)
- The user account that ran it
- The command line arguments (if command line auditing is enabled)

This makes 4688 invaluable for detecting tools like `mimikatz.exe`, `net.exe`, `whoami.exe`, `psexec.exe` being run by unexpected users.

**Event ID 1102 — Log Cleared:**  
An attacker who clears the Security log to hide their tracks will inadvertently generate this event — but only if a SIEM or log forwarding system has already captured the prior events. This is why **real-time log forwarding** to an external system is critical.

---

### 3.3 System Event IDs — Service Activity

These events live in the **System** log and track Windows Service lifecycle events — highly relevant given the persistence techniques covered in the Scheduled Tasks module.

| Event ID | Description | Why It Matters |
|---|---|---|
| **7045** | A **new service was installed** on the system | Direct correlation to `sc create` — any new service installation should be reviewed |
| **7030** | A service is **marked as an interactive service** | Interactive services are unusual in modern Windows and may indicate malicious configuration |
| **7035** | A service **control request was sent** (start/stop signal sent) | Tracks who is issuing service control commands |
| **7036** | A service **entered the running or stopped state** | Logs state transitions — correlate with 7045 to track newly installed services being started |

> **Correlation tip:** When investigating a suspected malicious service, chain these IDs together: **7045** (installed) → **7036** (started running) → **7036** (stopped) to reconstruct the full timeline.

## 4. Custom Views and Log Filtering in Event Viewer

Event Viewer can contain thousands of entries. Custom Views and filters let you focus on what matters.

### Creating a Custom View

1. Open **Event Viewer** → right-click **Custom Views** → **Create Custom View**
2. Set the time range (Last hour, Last 24 hours, custom range)
3. Choose **By log** (e.g., Security) or **By source**
4. Enter specific **Event IDs** (comma-separated, e.g., `4720, 4726, 4688`)
5. Optionally filter by **Event level** (Error, Warning, etc.) or **User/Computer**
6. Name the view and save — it persists across sessions

### Filtering Current Log

Right-click any log → **Filter Current Log** → same options as above, but temporary (not saved as a custom view).

> **Tip:** Creating a custom view for `4720,4726,4688,1102,7045` gives you a fast "suspicious activity dashboard" without any additional tooling.

## 6. Querying Event Logs from the Command Line

### 6.1 `wevtutil` — Windows Event Utility

`wevtutil` is the built-in Windows command-line tool for querying, exporting, and managing event logs.

**Query all events in the Security log:**

```cmd
wevtutil qe security
```

Returns all events from the Security log (can be very large — use filters).

---

**Query the last 5 events, formatted as text, in reverse order (newest first):**

```cmd
wevtutil qe security /c:5 /f:text /rd:true
```

| Flag | Meaning |
|---|---|
| `/c:5` | Return only the last **5 events** (count) |
| `/f:text` | Format output as **human-readable text** (default is XML) |
| `/rd:true` | **Reverse direction** — reads from newest to oldest |

---

**Filter for a specific Event ID (e.g., 4720 — account creation):**

```cmd
wevtutil qe security /c:5 /f:text /rd:true /q:"*[System[(EventID=4720)]]"
```

| Flag | Meaning |
|---|---|
| `/q:"..."` | XPath query to **filter events** — this is the standard Windows event filter syntax |
| `*[System[(EventID=4720)]]` | XPath expression: return events where the EventID equals 4720 |

**More XPath filter examples:**

```cmd
# Filter by multiple Event IDs
/q:"*[System[(EventID=4720 or EventID=4726)]]"

# Filter by Event ID AND time range (last 24 hours)
/q:"*[System[(EventID=4688) and TimeCreated[timediff(@SystemTime) <= 86400000]]]"
```

> **Note:** `wevtutil` XPath queries follow the **XPath 1.0** subset supported by Windows. The `timediff` value is in milliseconds (86400000 = 24 hours).

📚 Full reference: [wevtutil — Microsoft Learn](https://learn.microsoft.com/en-us/windows-server/administration/windows-commands/wevtutil)

### 6.2 PowerShell — `Get-WinEvent`

`Get-WinEvent` is the modern PowerShell cmdlet for querying Windows event logs. It is more flexible and scriptable than `wevtutil`.

**Query all events in the System log:**

```powershell
Get-WinEvent -LogName System
```

**Query the last 10 events from the Security log:**

```powershell
Get-WinEvent -LogName Security -MaxEvents 10
```

**Filter by Event ID using FilterHashtable (recommended approach):**

```powershell
Get-WinEvent -FilterHashtable @{
    LogName   = 'Security'
    Id        = 4720
}
```

**Filter by multiple Event IDs:**

```powershell
Get-WinEvent -FilterHashtable @{
    LogName = 'Security'
    Id      = @(4720, 4726, 4688, 1102)
}
```

**Filter by Event ID and time range:**

```powershell
Get-WinEvent -FilterHashtable @{
    LogName   = 'Security'
    Id        = 4688
    StartTime = (Get-Date).AddDays(-1)   # last 24 hours
}
```

**Format output for readability:**

```powershell
Get-WinEvent -FilterHashtable @{ LogName = 'Security'; Id = 4720 } |
    Select-Object TimeCreated, Id, Message |
    Format-List
```

**Export results to CSV for analysis:**

```powershell
Get-WinEvent -FilterHashtable @{ LogName = 'Security'; Id = 4720 } |
    Select-Object TimeCreated, Id, Message |
    Export-Csv -Path C:\events_4720.csv -NoTypeInformation
```

> **FilterHashtable vs `-LogName`:** Always prefer `-FilterHashtable` for performance — it filters at the log API level rather than pulling all events into memory first.

📚 Full reference:
- [Get-WinEvent — Microsoft Learn](https://learn.microsoft.com/en-us/powershell/module/microsoft.powershell.diagnostics/get-winevent?view=powershell-7.4)
- [Creating Get-WinEvent Queries with FilterHashtable](https://learn.microsoft.com/en-us/powershell/scripting/samples/creating-get-winevent-queries-with-filterhashtable?view=powershell-7.4)

## 7. Summary — Key Event IDs Cheat Sheet

### Security Log

| Event ID | Event | Red Flag Scenario |
|---|---|---|
| 4624 | Successful logon | Logon Type 10 (RDP) from unexpected IP |
| 4625 | Failed logon | Repeated failures = brute force |
| 4688 | Process created | `mimikatz`, `net.exe`, `whoami` run by unexpected user |
| 4720 | User account created | Any new account not created by IT |
| 4722 | User account enabled | Dormant account suddenly re-enabled |
| 4723 | Password change (by user) | Unexpected self-change |
| 4724 | Password reset (by admin) | Non-admin performing reset |
| 4725 | User account disabled | |
| 4726 | User account deleted | Account created then deleted quickly |
| 4738 | User account changed | Group membership or flag changes |
| 1102 | Security log cleared | **Always critical** — investigate immediately |

### System Log

| Event ID | Event | Red Flag Scenario |
|---|---|---|
| 7045 | New service installed | Service path in `C:\Users\` or `%TEMP%` |
| 7030 | Interactive service marked | Unusual in modern Windows |
| 7035 | Service control request sent | Unexpected user issuing start/stop |
| 7036 | Service state changed | New malicious service going from stopped → running |
